SECRETS IN KUBERNETES - DETAILED NOTES

1. Definition
- Secret is a Kubernetes object.
- It stores sensitive key-value data.
- Common uses: passwords, tokens, API keys, certificates.

2. Why Secrets exist
- Applications need configuration.
- Some configuration is sensitive.
- Hardcoding secrets in source code, images, or YAML is bad practice.
- Secrets separate sensitive config from application definitions.

3. Problems Secrets solve
- Prevent credentials from being hardcoded.
- Allow same image across multiple environments with different credentials.
- Provide standard integration with Pods.
- Organize sensitive data separately from normal config.

4. Important security reality
- Secret values are base64-encoded, not encrypted, by default.
- Base64 is reversible.
- Real security depends on:
  - RBAC
  - API server access control
  - etcd encryption at rest
  - node security
  - audit logging

5. Internal architecture flow
- User creates Secret through kubectl/YAML.
- Request goes to kube-apiserver.
- API server stores Secret in etcd.
- Pod references Secret.
- kubelet on node fetches Secret from API server.
- kubelet injects Secret into container as env vars or mounted files.

6. Secret manifest structure
- apiVersion: v1
- kind: Secret
- metadata
- type
- data or stringData

7. Secret type
- Default for generic secrets: Opaque
- Other important types:
  - kubernetes.io/tls
  - kubernetes.io/dockerconfigjson

8. data vs stringData
- data:
  - base64-encoded values required
- stringData:
  - plain text values allowed
  - API server converts them into data

9. Ways to consume Secrets in Pods
A. env
- Use one specific key via secretKeyRef

B. envFrom
- Import all keys from the Secret as environment variables

C. volume mount
- Mount Secret as files
- Each key becomes a file

10. Secret update behavior
- Secret mounted as volume:
  - may refresh on mounted files after some delay
- Secret used as env vars:
  - does not dynamically update
  - Pod restart is usually required

11. Imperative creation
- From literal:
  kubectl create secret generic app-secret --from-literal=username=admin --from-literal=password=pass123

- From file:
  kubectl create secret generic app-secret --from-file=config.txt

- From env file:
  kubectl create secret generic app-secret --from-env-file=secret.env

12. Declarative creation
- Use Secret YAML
- Prefer stringData for easier manifest authoring

13. Why generic is important
- Command format is:
  kubectl create secret <subtype> <name> ...
- subtype can be:
  - generic
  - tls
  - docker-registry
- generic means:
  - general-purpose secret
  - arbitrary key-value pairs
  - supports --from-literal, --from-file, --from-env-file
- generic usually results in type: Opaque

14. generic vs Opaque
- generic = kubectl CLI subcommand
- Opaque = Secret type stored in Kubernetes object

15. Specialized secret creation
- TLS Secret:
  kubectl create secret tls my-tls --cert=tls.crt --key=tls.key
- Docker registry Secret:
  kubectl create secret docker-registry regcred --docker-server=... --docker-username=... --docker-password=... --docker-email=...

16. Inspection commands
- kubectl get secret
- kubectl describe secret <name>
- kubectl get secret <name> -o yaml
- decode:
  kubectl get secret <name> -o jsonpath='{.data.key}' | base64 --decode

17. Best practices
- Store only sensitive data in Secrets
- Do not put secrets directly in Git
- Use RBAC carefully
- Enable etcd encryption at rest
- Prefer mounted files for rotating credentials/certs
- Use immutable secrets where suitable

18. Common exam mistakes
- forgetting generic
- writing plain text in data
- wrong capitalization of kind: Secret
- confusing ConfigMap and Secret
- expecting env vars to auto-refresh

19. CKA exam expectation
- Create Secret imperatively
- Create Secret declaratively
- Mount/use Secret in Pod
- Troubleshoot wrong Secret reference
- Understand difference from ConfigMap
- Understand importance of generic
