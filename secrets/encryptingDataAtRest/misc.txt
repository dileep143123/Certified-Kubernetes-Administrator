Encrypting Secret Data at Rest — misc.txt
========================================

FAST REVISION
-------------
- Secret.data is base64, not encryption
- Secrets are unencrypted in etcd by default
- kube-apiserver handles encryption/decryption
- EncryptionConfiguration decides which resources/providers are used
- first provider encrypts new writes
- providers/keys are tried in order for decryption
- old Secrets are not auto-encrypted
- rewrite existing Secrets to apply encryption
- identity often stays temporarily during migration
- KMS v2 is preferred where feasible

BACKEND STORY IN SIMPLE WORDS
-----------------------------
User -> kube-apiserver -> encryption provider -> etcd

Read:
etcd -> kube-apiserver -> decrypt -> client

So etcd does not “understand Secrets”.
It just stores bytes.
The intelligence sits in kube-apiserver.

LAB MENTAL MODEL
----------------
Before encryption:
etcd has plaintext Secret storage

After enabling encryption:
new Secret writes become encrypted in etcd

After rewrite:
old Secrets also become encrypted

VERY IMPORTANT DIFFERENCES
--------------------------
base64 vs encryption:
- base64 = representation
- encryption = confidentiality protection

at rest vs in transit:
- at rest = stored data
- in transit = network data

cluster secret encryption vs app data encryption:
- cluster secret encryption = Kubernetes API objects in etcd
- app data encryption = your DB/files/volumes handled separately

COMMON INTERVIEW / EXAM LINES
-----------------------------
Q: Why do we encrypt Secrets at rest?
A: To protect sensitive data if etcd or backups are accessed.

Q: Which component does the work?
A: kube-apiserver.

Q: Why rewrite old Secrets?
A: Because enabling encryption only affects new writes.

Q: Why not remove old key immediately?
A: Old data may still depend on it for decryption.

SHORT COMMAND MEMORY
--------------------
Config flag:
--encryption-provider-config=<path>

Optional reload:
--encryption-provider-config-automatic-reload=true

Rewrite old Secrets:
kubectl get secrets --all-namespaces -o json | kubectl replace -f -

WHAT TO PRACTICE
----------------
1. Create a Secret
2. Enable at-rest encryption for secrets
3. Create a new Secret
4. Verify encrypted prefix in etcd
5. Rewrite old Secrets
6. Rotate key safely
7. Verify reads still work

ONE EXAM-SAFE EXPLANATION
-------------------------
Encrypting Secret data at rest means kube-apiserver encrypts Secret objects before storing them in etcd. This protects stored data and backups from plaintext exposure. The first provider encrypts new writes, decryption tries providers and keys in order, and older objects must be rewritten to get encrypted.

STUDY WARNING
-------------
Do not memorize only commands.
Understand:
- who encrypts
- when encryption happens
- why old objects stay old
- why provider order matters
- why key rotation is multi-step
