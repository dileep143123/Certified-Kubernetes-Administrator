Encrypting Secret Data at Rest — questions.txt
=============================================

1) What does “encrypting Secret data at rest” mean?
Answer:
It means Kubernetes encrypts Secret object data before storing it persistently, mainly in etcd, so raw stored data is not readable in plaintext.

2) Is base64 encoding the same as encryption?
Answer:
No. Base64 is only encoding, not encryption. Anyone can decode base64 easily.

3) By default, how are Secrets stored in etcd?
Answer:
By default, Secret objects are stored unencrypted in etcd unless encryption at rest is configured.

4) Which component performs the encryption and decryption for Secret data at rest?
Answer:
kube-apiserver performs encryption before writing to etcd and decryption when reading from etcd.

5) Where is encryption at rest configured?
Answer:
On kube-apiserver using an EncryptionConfiguration file and the --encryption-provider-config flag.

6) Which provider is used to encrypt new writes?
Answer:
The first provider in the providers list is used to encrypt new writes.

7) How does decryption work?
Answer:
Providers and keys are tried in order until kube-apiserver can decrypt the stored data.

8) What does the identity provider do?
Answer:
It does not encrypt data. It stores or reads data as plaintext.

9) Why is identity sometimes left in the provider list during migration?
Answer:
So old plaintext objects can still be read while new writes are encrypted.

10) Does enabling encryption automatically encrypt old existing Secrets?
Answer:
No. Existing Secrets remain in their old storage form until they are rewritten.

11) How can old Secrets be re-encrypted after enabling encryption?
Answer:
By rewriting them through the API, for example:
kubectl get secrets --all-namespaces -o json | kubectl replace -f -

12) What does a stored prefix like k8s:enc:aescbc:v1:key1 indicate?
Answer:
It indicates the data stored in etcd is encrypted using the aescbc provider and the named key.

13) What is the danger of removing an old key too early?
Answer:
Objects encrypted with that old key may become unreadable.

14) What is the difference between encryption at rest and encryption in transit?
Answer:
Encryption at rest protects stored data in etcd/backups.
Encryption in transit protects network communication, usually with TLS.

15) Does encryption at rest for Secrets automatically encrypt files mounted into Pods?
Answer:
No. Mounted filesystem data needs storage-level encryption or application-level encryption.

16) What is KMS used for?
Answer:
KMS is used for external key management and envelope encryption, commonly for stronger production security practices.

17) In modern Kubernetes guidance, which KMS version is preferred where feasible?
Answer:
KMS v2 is preferred where feasible.

18) Why must control plane nodes use consistent encryption configuration?
Answer:
Because every kube-apiserver must be able to decrypt the stored data safely during normal operation and rollout changes.

19) What is the core mental model for this topic?
Answer:
Write path uses the first provider.
Read path tries providers/keys in order.
Old objects need rewriting.

20) Give one-line CKA exam answer for this topic.
Answer:
Kubernetes Secret encryption at rest is configured on kube-apiserver so Secret data is encrypted before being stored in etcd; old Secrets must be rewritten to become encrypted.
