Encrypting Secret Data at Rest — notes.txt
=========================================

1) What this topic means
------------------------
“Encrypting Secret data at rest” means protecting Secret objects when Kubernetes stores them in persistent storage, mainly etcd.

Important distinction:
- At rest = data stored on disk / in etcd / in backups
- In transit = data moving over the network, usually protected with TLS

This topic is about the Kubernetes API server encrypting stored API object data before it is written to etcd.

2) Why this matters
-------------------
By default, Secret objects are stored unencrypted in etcd.
That means:
- anyone with direct etcd access
- anyone with access to an etcd snapshot / backup
may be able to read sensitive data.

Encryption at rest adds another protection layer so the raw stored bytes in etcd are encrypted.

3) What exactly gets encrypted
------------------------------
Kubernetes supports at-rest encryption for persistent API resource data.
Examples:
- Secrets
- ConfigMaps
- custom resources (on supported Kubernetes versions)

For the CKA exam, the most important example is Secrets.

4) What path the data takes
---------------------------
Typical flow:
1. User runs kubectl create/apply for a Secret
2. Request goes to kube-apiserver
3. kube-apiserver authenticates and authorizes the request
4. Before writing the object to etcd, kube-apiserver checks the EncryptionConfiguration
5. kube-apiserver uses the first matching provider in the provider list to encrypt the object for storage
6. Encrypted bytes are stored in etcd

On read:
1. kube-apiserver fetches stored bytes from etcd
2. it tries providers / keys in order until it can decrypt
3. it returns the normal Secret object back through the API

So:
- encryption/decryption happens at the API server layer
- clients still see the normal Secret through kubectl/API if they are authorized

5) Very important: base64 is NOT encryption
-------------------------------------------
Secret values in YAML often look unreadable because they are base64-encoded.
Base64 is only an encoding format.
It is easy to decode.
That is not the same as encrypting data at rest.

6) Where encryption is configured
---------------------------------
It is configured on kube-apiserver using:
- an EncryptionConfiguration file
- and the flag:
  --encryption-provider-config=<path>

Optional automatic reload:
  --encryption-provider-config-automatic-reload=true

7) How EncryptionConfiguration works
------------------------------------
The config maps:
- resources to encrypt
- providers to use

The provider list is ordered.

Rule:
- the first provider is used for encryption of new writes
- providers/keys are tried in order for decryption

This is why order matters a lot.

8) Main providers you should know
---------------------------------
identity:
- no encryption
- stores data as plaintext
- default if nothing else is configured

aescbc:
- common exam-friendly provider
- symmetric encryption with local key material in config

secretbox:
- another local encryption option

kms:
- envelope encryption using an external Key Management Service
- better for production-grade setups
- modern recommendation is KMS v2 where feasible

For CKA-style understanding, aescbc is the easiest to reason about in labs.
In modern production guidance, KMS v2 is preferred where feasible.

9) Mental model of providers
----------------------------
Think of the provider chain like this:

Write path:
- first provider wins for encryption

Read path:
- kube-apiserver attempts decryption with providers/keys in order

That allows migration and key rotation without immediate downtime, as long as old decryption capability remains available.

10) Why identity is often kept temporarily
------------------------------------------
During migration, administrators often keep:
- encryption provider first
- identity later

Why?
Because:
- new writes get encrypted
- old plaintext objects can still be read

After all old objects are rewritten and encrypted, identity can be removed to prevent accidental plaintext reads.

11) Why existing Secrets may remain plaintext
---------------------------------------------
Enabling encryption does NOT automatically rewrite old Secret objects already stored in etcd.

Only new writes use the first encryption provider.

So existing objects must be rewritten to become encrypted at rest.

Classic rewrite command:
kubectl get secrets --all-namespaces -o json | kubectl replace -f -

This reads and writes them back so the API server stores them using the active encryption provider.

12) How to verify it actually works
-----------------------------------
There are two kinds of verification:

A. Logical verification
- kubectl get secret ... still works normally
- this proves API server can decrypt and serve the object

B. Storage verification
- inspect etcd stored bytes
- encrypted objects show a prefix like:
  k8s:enc:aescbc:v1:key1

That prefix indicates the chosen provider and key name were used.

13) Key rotation concept
------------------------
Key rotation means:
- add a new key
- keep old key temporarily for decryption
- make new key first in the key list so new writes use it
- rewrite existing objects
- remove old key only after everything is safely re-encrypted

If you remove the old key too early, previously encrypted objects may become unreadable.

14) Automatic reload
--------------------
If automatic reload is enabled:
--encryption-provider-config-automatic-reload=true

Then kube-apiserver can detect changes to the encryption config file without restart.
This helps during key rotation.

15) KMS high-level backend view
-------------------------------
KMS provider uses envelope encryption.

Mental model:
- DEK = data encryption key
- KEK = key encryption key

The Secret data is encrypted with a DEK.
That DEK is protected by a KEK managed in an external KMS.

This reduces direct exposure of root encryption keys inside Kubernetes.

16) At rest vs in transit
-------------------------
At rest:
- protects stored API data in etcd / backups

In transit:
- protects communication over the network using TLS

Both are important.
Encrypting at rest does NOT replace TLS.
TLS does NOT replace encryption at rest.

17) What this does NOT cover
----------------------------
This feature does NOT encrypt:
- random files inside container filesystems
- application data in your own app database
- data on volumes unless the storage layer itself encrypts it

For mounted storage, use:
- storage-level encryption
or
- application-level encryption

18) Exam-focused key takeaways
------------------------------
Remember these points:
- Secrets are unencrypted in etcd by default
- base64 is not encryption
- kube-apiserver performs at-rest encryption
- EncryptionConfiguration controls resources/providers
- first provider encrypts new writes
- providers/keys are tried in order for decryption
- old Secrets are not auto-encrypted; rewrite them
- identity is often kept during migration and removed later
- KMS v2 is preferred in modern production where feasible

19) Common mistakes
-------------------
- thinking Secret.data base64 means secure
- enabling encryption config and assuming all old Secrets are already encrypted
- removing old keys too early
- forgetting same config must exist across control plane nodes
- confusing encryption at rest with TLS in transit
- thinking this encrypts app data stored in volumes or databases automatically

20) Short final summary
-----------------------
Encrypting Secret data at rest means kube-apiserver encrypts Secret objects before storing them in etcd.
It protects against etcd/backup exposure.
The first provider encrypts new writes, decryption tries providers/keys in order, and old objects need rewriting to become encrypted.
