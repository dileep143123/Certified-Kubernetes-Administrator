SECRETS - EXAM TIPS, SHORTCUTS, TRICKS

1. Most important imperative command
kubectl create secret generic app-secret \
  --from-literal=username=admin \
  --from-literal=password=pass123

2. Dry-run YAML generation
kubectl create secret generic app-secret \
  --from-literal=username=admin \
  --from-literal=password=pass123 \
  --dry-run=client -o yaml

3. Create from env file
kubectl create secret generic app-secret --from-env-file=secret.env

4. Create from file
kubectl create secret generic app-secret --from-file=config.txt

5. Decode a secret value
kubectl get secret app-secret -o jsonpath='{.data.username}' | base64 --decode
echo

6. Remember this mapping
generic -> Opaque
tls -> kubernetes.io/tls
docker-registry -> kubernetes.io/dockerconfigjson

7. Super important memory trick
ConfigMap = normal config
Secret = sensitive config

8. Another memory trick
stringData = human-friendly input
data = base64-encoded stored form

9. Pod env usage pattern
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: app-secret
      key: password

10. Pod envFrom usage pattern
envFrom:
- secretRef:
    name: app-secret

11. Pod volume usage pattern
volumes:
- name: secret-vol
  secret:
    secretName: app-secret

volumeMounts:
- name: secret-vol
  mountPath: /etc/secret
  readOnly: true

12. CKA trap
If the question says create a secret from literal:
Use:
kubectl create secret generic ...
Not:
kubectl create secret ...

13. Another CKA trap
If writing YAML manually, avoid manually encoding values unless needed.
Use stringData for speed.

14. Debugging checklist if Pod is not getting Secret
- Does the Secret exist?
- Is it in the same namespace?
- Did you use the correct secret name?
- Did you use the correct key name?
- Did container spec reference it correctly?
- Did Pod restart if using env vars?
- Is Secret mounted at expected path?

15. Namespace rule
Pod and Secret generally must be in the same namespace.

16. Fast exam workflow
- create imperatively
- verify with kubectl get secret
- patch or regenerate YAML if needed
- reference in Pod
- exec into Pod and verify presence

17. Verification command for mounted secret
kubectl exec -it <pod-name> -- ls /etc/secret
kubectl exec -it <pod-name> -- cat /etc/secret/password

18. Verification command for env secret
kubectl exec -it <pod-name> -- printenv | grep PASSWORD

19. Security reminder
Base64 != encryption

20. One-line exam summary
Secret is for sensitive key-value data; generic creates arbitrary Opaque secrets; kubelet injects them into Pods via env vars or mounted files.
