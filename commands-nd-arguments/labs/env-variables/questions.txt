1. What is an environment variable in the context of a containerized application?
An environment variable is a key-value pair provided to the application process at startup. Inside a container, the application can read these values to determine runtime configuration such as database host, log level, or mode.

2. Where are environment variables defined in a Kubernetes Pod manifest?
They are defined under:
spec.containers[].env
or
spec.containers[].envFrom

3. Are environment variables configured at the Pod level or container level?
They are configured at the container level. Each container in a Pod has its own environment variable configuration.

4. What is the difference between env and envFrom in Kubernetes?
env allows you to define variables one by one, including literals or specific key references from ConfigMaps/Secrets.
envFrom imports all keys from a ConfigMap or Secret as environment variables.

5. When should you use env instead of envFrom?
Use env when you want only selected keys, want to rename a variable, or want precise control over each variable.

6. When should you use envFrom?
Use envFrom when you want to load many keys from a ConfigMap or Secret at once and are okay using the same key names as variable names.

7. What is a ConfigMap used for?
A ConfigMap stores non-sensitive configuration data such as application mode, port numbers, feature flags, and service endpoints.

8. What is a Secret used for?
A Secret stores sensitive data such as passwords, API keys, access tokens, and credentials.

9. How do you inject a single key from a ConfigMap as an environment variable?
By using:
valueFrom:
  configMapKeyRef:
    name: <configmap-name>
    key: <key-name>

10. How do you inject a single key from a Secret as an environment variable?
By using:
valueFrom:
  secretKeyRef:
    name: <secret-name>
    key: <key-name>

11. If a ConfigMap value is used as an environment variable and the ConfigMap is updated, will the running Pod automatically see the new value?
No. Environment variables are injected when the container starts. A restart or rollout is required for the new value to be seen.

12. Why do running containers not automatically get updated environment variable values from ConfigMap changes?
Because environment variables belong to the process environment created at process startup. Kubernetes does not mutate the process environment of a running container.

13. What is the Downward API?
The Downward API allows a container to access information about itself or its Pod, such as Pod name, namespace, node name, and Pod IP.

14. Give examples of metadata that can be injected using fieldRef.
metadata.name
metadata.namespace
spec.nodeName
status.podIP

15. What is resourceFieldRef used for?
It is used to expose container resource requests or limits as environment variables.

16. What is the difference between command/args and environment variables?
command/args define what process to run and with what CLI options.
Environment variables provide runtime configuration that the application reads internally.

17. How can you verify environment variables inside a running container?
Using:
kubectl exec -it <pod-name> -- env
or
kubectl exec -it <pod-name> -- printenv

18. How can you see environment variable configuration in the resource definition?
Using:
kubectl get pod <pod-name> -o yaml
or
kubectl describe pod <pod-name>

19. Does a Secret value remain base64 encoded inside the container?
No. It is decoded before being presented to the application as an environment variable.

20. What is a common YAML mistake while setting environment variables?
Placing env directly under spec instead of under spec.containers[]. Another common mistake is wrong indentation.

21. Can two containers in the same Pod automatically share environment variables?
No. Each container has its own environment section. If both need the same variable, both must be configured.

22. What happens to a Deployment when you update an environment variable in its Pod template?
The Pod template changes, which triggers a new rollout and new Pods are created.

23. Why is quoting values like "true" or "8080" often recommended in YAML?
Because environment variable values are strings and quoting avoids YAML type interpretation issues.

24. How can you quickly create a ConfigMap from the command line for CKA tasks?
kubectl create configmap app-config --from-literal=APP_COLOR=blue --from-literal=APP_MODE=prod

25. How can you quickly create a Secret from the command line for CKA tasks?
kubectl create secret generic app-secret --from-literal=DB_PASSWORD=password

26. What happens if envFrom imports keys that are not valid environment variable names?
Those keys may not be usable as valid environment variables for the application. In practice, keys should follow env var naming conventions.

27. What is the advantage of using ConfigMap/Secret instead of hardcoding env values in YAML?
It separates configuration from workload definition, improves reusability, and makes changes easier to manage.

28. How would you restart a Deployment so new environment variable values take effect?
kubectl rollout restart deployment <deployment-name>

29. Is base64 encoding in Secret equal to encryption?
No. Base64 is only encoding, not encryption.

30. In the exam, what is the fastest way to troubleshoot an application not receiving an environment variable?
Check the resource YAML, describe the Pod, verify referenced ConfigMap/Secret exists and has the correct key, then exec into the container and run env or printenv.
